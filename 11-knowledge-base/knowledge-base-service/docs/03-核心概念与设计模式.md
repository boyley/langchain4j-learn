# 核心概念与设计模式

## 一、核心概念

### 1. Embedding (嵌入/向量化)

```
┌─────────────────────────────────────────────────────────────┐
│                    什么是 Embedding?                         │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│   人类语言                        计算机数学                   │
│                                                             │
│   "如何请假"      ───────────→    [0.12, 0.34, 0.56, ...]    │
│                   EmbeddingModel        1536个数字            │
│                                                             │
│   为什么需要?                                                 │
│   - 计算机不能直接比较 "如何请假" 和 "年假规定" 是否相关         │
│   - 转成数字向量后，可以用数学方法计算相似度                     │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

**代码位置**: `config/EmbeddingModelConfig.java`

```java
@Bean
public EmbeddingModel embeddingModel() {
    return OpenAiEmbeddingModel.builder()
            .apiKey(apiKey)
            .modelName("text-embedding-3-small")  // 输出 1536 维向量
            .build();
}
```

### 2. 向量存储 (Vector Store)

```
┌─────────────────────────────────────────────────────────────┐
│                    什么是向量存储?                            │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│   普通数据库 (MySQL)              向量数据库 (Redis/Milvus)    │
│                                                             │
│   SELECT * FROM docs             SELECT * FROM vectors       │
│   WHERE title = '请假'           WHERE vector ≈ [0.1,0.2,.]  │
│        ↑                              ↑                      │
│     精确匹配                       相似度搜索 (KNN)            │
│                                                             │
│   向量数据库的核心能力:                                        │
│   - 存储高维向量 (1536维)                                     │
│   - 快速找到最相似的向量 (毫秒级)                              │
│   - 支持百万/亿级数据                                         │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

**代码位置**: `store/VectorStoreService.java` (接口)

```java
public interface VectorStoreService {
    // 存储: 向量 + 原文
    String store(Embedding embedding, TextSegment segment);

    // 搜索: 输入查询向量，返回最相似的结果
    List<EmbeddingMatch<TextSegment>> search(
        Embedding queryEmbedding, int maxResults, double minScore);

    // 删除
    int deleteByDocId(String docId);
}
```

### 3. 文档分割 (Document Splitting)

```
┌─────────────────────────────────────────────────────────────┐
│                    为什么要分割文档?                          │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│   原始文档 (5000字)                                          │
│   ┌────────────────────────────────────────────────────┐    │
│   │ 第一章 年假规定                                      │    │
│   │ 1. 工作满1年享有5天年假...                           │    │
│   │ 2. 年假可分次使用...                                 │    │
│   │                                                    │    │
│   │ 第二章 请假流程                                      │    │
│   │ 1. 提前3天在OA提交...                                │    │
│   │ ...                                                │    │
│   └────────────────────────────────────────────────────┘    │
│                         │                                   │
│                         ▼ 分割                              │
│   ┌──────────────┐ ┌──────────────┐ ┌──────────────┐       │
│   │ 片段1        │ │ 片段2        │ │ 片段3        │       │
│   │ 年假规定...  │ │ 请假流程...  │ │ 审批要求...  │       │
│   │ (500字)      │ │ (500字)      │ │ (500字)      │       │
│   └──────────────┘ └──────────────┘ └──────────────┘       │
│                                                             │
│   为什么:                                                    │
│   - 向量化有长度限制 (一般 8000 token)                        │
│   - 搜索返回的是片段，小片段更精准                             │
│   - 重叠保证上下文不丢失                                      │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

**代码位置**: `pipeline/KnowledgePipeline.java`

```java
// 创建分割器: 每段最多500字，重叠50字
private DocumentSplitter getSplitter() {
    return DocumentSplitters.recursive(segmentMaxSize, segmentOverlap);
    //                                      500           50
}
```

### 4. 增量同步 (Incremental Sync)

```
┌─────────────────────────────────────────────────────────────┐
│                    增量 vs 全量同步                          │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│   全量同步 (Full Sync)                                       │
│   - 每次处理所有文档                                         │
│   - 10000篇文档 = 调用10000次 OpenAI API                     │
│   - 慢，贵                                                  │
│                                                             │
│   增量同步 (Incremental Sync)                                │
│   - 只处理"上次同步后修改过的"文档                            │
│   - 10000篇中只有10篇更新 = 只调用10次 API                    │
│   - 快，省钱                                                │
│                                                             │
│   实现方式:                                                  │
│   - 记录上次同步时间 (sync_status 表)                         │
│   - 查询时 WHERE update_time > last_sync_time               │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

**代码位置**: `source/impl/DatabaseKnowledgeSource.java`

```java
public List<KnowledgeDocument> fetchIncremental(LocalDateTime lastSyncTime) {
    if (lastSyncTime == null) {
        return documentRepository.findByStatus(1);  // 全量
    }
    // 只查更新过的
    return documentRepository.findByStatusAndUpdateTimeAfter(1, lastSyncTime);
}
```

## 二、设计模式

### 1. 策略模式 (Strategy Pattern)

**用在哪**: 向量存储、知识源

```
┌─────────────────────────────────────────────────────────────┐
│                       策略模式                               │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│   VectorStoreService (接口)                                  │
│          ▲                                                  │
│          │ implements                                       │
│   ┌──────┴──────┬──────────────┐                           │
│   │             │              │                           │
│   ▼             ▼              ▼                           │
│ Redis实现    PgVector实现    Milvus实现                      │
│                                                             │
│   好处:                                                      │
│   - 切换向量库只需改配置，不改业务代码                          │
│   - 新增向量库实现只需新增一个类                               │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

**配置切换** (application.yml):
```yaml
knowledge:
  vector-store:
    type: redis      # 改成 pgvector 就用 PostgreSQL
```

**Spring 条件加载**:
```java
@Service
@ConditionalOnProperty(name = "knowledge.vector-store.type", havingValue = "redis")
public class RedisVectorStoreService implements VectorStoreService {
    // 只有配置 type=redis 时才创建这个 Bean
}
```

### 2. 管道模式 (Pipeline Pattern)

**用在哪**: 知识处理流程

```
┌─────────────────────────────────────────────────────────────┐
│                       管道模式                               │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│   输入                                                       │
│     │                                                       │
│     ▼                                                       │
│ ┌─────────┐    ┌─────────┐    ┌─────────┐    ┌─────────┐  │
│ │  加载   │───→│  分割   │───→│ 向量化  │───→│  存储   │  │
│ │ 文档    │    │ 片段    │    │ 嵌入    │    │ 向量库  │  │
│ └─────────┘    └─────────┘    └─────────┘    └─────────┘  │
│                                                             │
│   每一步职责单一，可独立测试和替换                             │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

**代码位置**: `pipeline/KnowledgePipeline.java`

```java
public int processDocument(KnowledgeDocument doc) {
    // Step 1: 删除旧向量
    vectorStore.deleteByDocId(doc.getDocId());

    // Step 2: 构建 Document
    Document document = Document.from(doc.getContent(), metadata);

    // Step 3: 分割
    List<TextSegment> segments = getSplitter().split(document);

    // Step 4: 向量化 + 存储
    for (TextSegment segment : segments) {
        Embedding embedding = embeddingModel.embed(segment.text()).content();
        vectorStore.store(embedding, segment);
    }

    return segments.size();
}
```

## 三、数据库设计

### 企业级设计: 内容与元数据分离

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                    为什么要分离存储内容?                                     │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│   问题场景:                                                                  │
│   SELECT * FROM knowledge_documents WHERE status = 1                        │
│                                                                             │
│   旧设计 (content 在主表):                                                   │
│   ┌────────────────────────────────────────────────────────────────────┐   │
│   │ 返回 1000 条记录，每条含 1MB content                                │   │
│   │ 内存占用: 1000 × 1MB = 1GB                                         │   │
│   │ 结果: OOM!                                                         │   │
│   └────────────────────────────────────────────────────────────────────┘   │
│                                                                             │
│   新设计 (content 分表):                                                    │
│   ┌────────────────────────────────────────────────────────────────────┐   │
│   │ Step 1: 查元数据，1000 条 × 1KB = 1MB                              │   │
│   │ Step 2: 逐个处理时，每次只加载 1 条的 content                       │   │
│   │ 结果: 内存始终控制在可接受范围                                       │   │
│   └────────────────────────────────────────────────────────────────────┘   │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

### 表结构

```sql
-- 知识文档元数据表 (轻量级)
CREATE TABLE knowledge_documents (
    id BIGINT PRIMARY KEY,
    doc_id VARCHAR(128) UNIQUE,      -- 文档唯一标识
    title VARCHAR(255),              -- 标题
    summary VARCHAR(500),            -- 摘要 (前500字符，用于预览)
    category VARCHAR(64),            -- 分类
    source VARCHAR(32),              -- 来源: database/file

    -- 内容元信息 (不存储实际内容!)
    content_size BIGINT,             -- 内容大小 (字节)
    content_hash VARCHAR(64),        -- 内容哈希 (用于变更检测)

    vector_status INT DEFAULT 0,     -- 向量化状态
    segment_count INT,               -- 向量化片段数
    update_time DATETIME
);

-- 知识内容表 (分离存储大内容)
CREATE TABLE knowledge_content (
    id BIGINT PRIMARY KEY,
    doc_id VARCHAR(128) UNIQUE,      -- 关联文档ID
    content LONGTEXT,                -- 完整内容 (最大 4GB)
    content_size BIGINT,
    content_hash VARCHAR(64),
    FOREIGN KEY (doc_id) REFERENCES knowledge_documents(doc_id)
);

-- 同步状态表
CREATE TABLE sync_status (
    id BIGINT PRIMARY KEY,
    source_name VARCHAR(64) UNIQUE,
    last_sync_time DATETIME,
    last_sync_status VARCHAR(32),
    last_sync_count INT
);
```

### 数据流向

```
┌───────────────────────────────────────────────────────────────────────────────┐
│                              数据存储位置                                      │
├───────────────────────────────────────────────────────────────────────────────┤
│                                                                               │
│   MySQL                                           Redis                       │
│   ┌─────────────────────────────────────┐        ┌─────────────────────────┐ │
│   │ knowledge_documents (元数据)         │        │ knowledge_vectors       │ │
│   │ - doc_id, title, category           │        │ - embedding (1536维)    │ │
│   │ - content_size, content_hash        │        │ - text (片段原文)       │ │
│   │ - vector_status                     │        │ - metadata              │ │
│   │                                     │  同步   │                         │ │
│   │ 查询快，内存小                       │ ─────→ │ 用途: 语义搜索          │ │
│   └─────────────────────────────────────┘        └─────────────────────────┘ │
│                    │                                                          │
│                    │ 1:1 关联                                                 │
│                    ▼                                                          │
│   ┌─────────────────────────────────────┐                                    │
│   │ knowledge_content (内容)             │                                    │
│   │ - doc_id                            │                                    │
│   │ - content (LONGTEXT)                │                                    │
│   │                                     │                                    │
│   │ 按需加载，可能很大                   │                                    │
│   └─────────────────────────────────────┘                                    │
│                                                                               │
└───────────────────────────────────────────────────────────────────────────────┘
```

### 代码使用示例

```java
// 批量查询 - 只加载元数据，不含内容
List<KnowledgeDocument> docs = documentRepository.findByStatus(1);
// docs 很轻量，可以安全地加载 10000 条

// 处理时按需加载内容
for (KnowledgeDocument doc : docs) {
    // 通过 ContentService 单独加载这一条的内容
    String content = contentService.getContent(doc.getDocId())
            .orElseThrow(() -> new RuntimeException("内容不存在"));

    // 处理这条文档...
    processDocument(doc, content);

    // 处理完后 content 可被 GC 回收
}

## 四、配置说明

### application.yml 关键配置

```yaml
knowledge:
  # 嵌入模型 - 把文字转成向量
  embedding:
    provider: openai
    openai:
      api-key: ${OPENAI_API_KEY}       # 从环境变量读取
      model: text-embedding-3-small    # 1536 维

  # 向量存储 - 存储和搜索向量
  vector-store:
    type: redis                        # 可选: redis, pgvector, milvus
    dimension: 1536                    # 必须和模型维度匹配!

  # 知识源 - 知识从哪里来
  source:
    database:
      enabled: true                    # 从数据库读取
    file:
      enabled: false                   # 从文件系统读取
      root-path: /data/docs

  # 处理管道 - 如何处理知识
  pipeline:
    batch-size: 100                    # 每批处理多少文档
    segment-max-size: 500              # 每个片段最大字符
    segment-overlap: 50                # 片段重叠字符

  # 定时同步
  sync:
    incremental:
      enabled: true
      cron: "0 0 * * * *"              # 每小时增量同步
```

### 环境变量

| 变量名 | 必填 | 说明 |

| MYSQL_PASSWORD | 否 | MySQL 密码，默认 root |
